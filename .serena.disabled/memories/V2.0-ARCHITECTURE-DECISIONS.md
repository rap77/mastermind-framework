**Fecha:** 2026-03-13
**Estado:** Planning Complete, Ready for Phase 1

---

## Core Architectural Decisions

### 1. Paralelización: asyncio.TaskGroup (NOT Celery/RQ)

**Decision:** Use Python 3.11+ `asyncio.TaskGroup` for parallel brain execution

**Why:**
- Brains are I/O-bound (MCP calls, NotebookLM queries) not CPU-bound
- asyncio provides sufficient parallelism for single-host deployment
- Celery/RQ adds Redis/RabbitMQ complexity prematurely
- Defer distributed workers to v3.0 when multi-host is needed

**Trade-offs:**
- ✅ Simpler architecture, no external dependencies
- ✅ Native Python, better debugging
- ❌ Limited to single host (acceptable for v2.0)

**Implementation:**
- NetworkX for dependency DAG resolution
- Topological sort for execution levels
- Task state in SQLite (not Redis)

### 2. Type Safety: Pydantic v2 + mypy --strict

**Decision:** Enforce strict type safety across all components

**Why:**
- Catches errors at dev time, not runtime
- Enables IDE autocomplete and refactoring
- Pydantic validates at runtime too (defense in depth)
- Essential for web API contracts

**Trade-offs:**
- ✅ Cleaner, more maintainable code
- ✅ Self-documenting (types are docs)
- ❌ Initial migration effort (fixing all type errors)

**Implementation:**
- Phase 1: Add Pydantic models for ALL data structures
- Phase 1: Enable `mypy --strict` in CI
- Phase 1: Create type-safe MCP wrapper

### 3. Web UI: FastAPI + HTMX (NOT Streamlit/Dash)

**Decision:** Build web dashboard with FastAPI backend and HTMX/Alpine.js frontend

**Why:**
- Streamlit/Dash designed for data science notebooks, not orchestration UIs
- FastAPI provides native async + WebSocket + OpenAPI
- HTMX enables dynamic UI without SPA build complexity
- User wants multi-user session support (harder with Streamlit)

**Trade-offs:**
- ✅ Separation of concerns (API vs UI)
- ✅ Custom UX, not constrained by notebook paradigm
- ❌ More frontend work than Streamlit

**Implementation:**
- Phase 3: FastAPI REST + WebSocket routes
- Phase 3: HTMX/Alpine.js dashboard (mobile-responsive)
- Phase 3: Per-request orchestrator instances (session isolation)

### 4. Experience Storage: JSONB files (NOT PostgreSQL yet)

**Decision:** Store execution logs as JSONB files, with schema ready for pgvector

**Why:**
- Full RAG with vector DB is premature optimization for v2.0
- NotebookLM handles semantic search already
- JSONB files upgradable to PostgreSQL + pgvector in v3.0
- `embedding_stub` placeholder in schema for future migration

**Trade-offs:**
- ✅ Simpler deployment (no database setup)
- ✅ Easy migration path to v3.0
- ❌ Keyword-only search (acceptable for now)

**Implementation:**
- Phase 4: ExperienceRecord schema with `embedding_stub: str | None`
- Phase 4: JSONB file storage with keyword search
- v3.0: Migrate to PostgreSQL + pgvector

### 5. Session Isolation: Per-request Orchestrator Instances

**Decision:** Eliminate global state, create orchestrator per request/session

**Why:**
- Current coordinator uses instance variables (breaks multi-user)
- Web UI requires concurrent sessions
- Essential for agency use case (multiple clients)

**Trade-offs:**
- ✅ True multi-user support
- ✅ No cross-session pollution
- ❌ Refactoring effort (extract global state)

**Implementation:**
- Phase 3: Dependency injection for orchestrator creation
- Phase 3: Session-scoped task state (user_id in queries)
- Phase 4: Multi-user E2E tests

## Phase Dependencies

```
Phase 1 (Type Safety)
    ↓ enables
Phase 2 (Parallel Execution)
    ↓ enables
Phase 3 (Web UI)
    ↓ enables
Phase 4 (Production Hardening)
```

**Why this order:**
1. Type safety prevents rewriting parallel code later
2. Parallel execution must exist before visualizing it
3. Web UI needs typed API + parallel backend
4. Production hardening verifies all components work together
