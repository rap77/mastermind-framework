# Technical Patterns - JWT Auth in Next.js 16 + FastAPI

## Problem Statement
Verifying JWT tokens generated by Python FastAPI (python-jose) in Next.js 16 frontend (jose v6+) fails with signature verification errors.

## Root Cause Analysis
1. **Library Incompatibility:** python-jose and jose v6 use different internal implementations for HS256
2. **Next.js 16 Restriction:** Server Components cannot do fetch() during render (Performance API error)
3. **Secret Key Exposure:** MM_SECRET_KEY should not be in frontend bundle

## Solution Pattern

### Architecture
```
Frontend (Next.js)          Backend (FastAPI)
     Client Comp  ──────────────►  /api/auth/verify
          │                              │
          │  httpOnly cookie            │  python-jose.verify()
          │  (access_token)             │  (same library)
          ▼                              ▼
    Set on login                   Returns {valid, user_id}
```

### Implementation Steps

**1. Backend Endpoint (Python):**
```python
@router.get("/verify")
async def verify_token_endpoint(request: Request) -> dict[str, object]:
    token = request.cookies.get("access_token")
    if not token:
        return {"valid": False, "user_id": None}

    try:
        payload = jwt.decode(token, SECRET_KEY, algorithms=[ALGORITHM])
        user_id = payload.get("sub")
        return {"valid": True, "user_id": user_id}
    except JWTError:
        return {"valid": False, "user_id": None}
```

**2. Frontend Verification (TypeScript):**
```typescript
export async function verifyToken(token: string): Promise<boolean> {
  const apiUrl = process.env.NEXT_PUBLIC_API_URL || 'http://localhost:8001'
  const response = await fetch(`${apiUrl}/api/auth/verify`, {
    method: 'GET',
    credentials: 'include', // httpOnly cookies
    cache: 'no-store',
  })

  if (!response.ok) return false

  const data = await response.json()
  return data.valid === true
}
```

**3. Client Component Wrapper:**
```typescript
'use client'

export function AuthGuard({ children }: { children: React.ReactNode }) {
  const router = useRouter()
  const [isAuthenticated, setIsAuthenticated] = useState(false)

  useEffect(() => {
    async function verifyToken() {
      const response = await fetch('/api/auth/verify', {
        credentials: 'include',
      })

      if (response.ok) {
        const data = await response.json()
        if (data.valid) {
          setIsAuthenticated(true)
        } else {
          router.push('/login')
        }
      } else {
        router.push('/login')
      }
    }

    verifyToken()
  }, [router])

  if (!isAuthenticated) return <div>Loading...</div>

  return <>{children}</>
}
```

**4. Server Component Layout:**
```typescript
import { AuthGuard } from '@/components/auth/AuthGuard'

export default function ProtectedLayout({ children }: { children: React.ReactNode }) {
  return <AuthGuard>{children}</AuthGuard>
}
```

## Key Benefits
1. **Eliminates library incompatibility** - Backend uses same library for generate + verify
2. **More secure** - MM_SECRET_KEY never exposed to frontend
3. **Simpler frontend** - No crypto operations, just HTTP fetch
4. **Seamless cookies** - Works perfectly with httpOnly cookies

## Trade-offs
- **Additional network hop** - Frontend calls backend for verification (adds ~5-10ms)
- **Backend dependency** - Frontend cannot verify tokens offline (acceptable for web app)

## When to Use This Pattern
- Next.js 13+ with App Router
- FastAPI backend with JWT auth
- python-jose vs jose library incompatibility
- Server Components need auth verification
