## Brain-Informed Updates

### CRITICAL Changes from Brain #2 (UX Research)

**Miller's Law Violation (24-Brain Display)**
- **Problem:** 24 simultaneous status badges = 3.4x Miller's limit (7±2 items)
- **Fix:** Add domain grouping (Product Strategy brains, UX brains, etc.) with collapsible sections
- **Priority:** CRITICAL — core usability issue
- **Acceptance:** Brains grouped by domain, collapsible sections, default to "Active Only" view
- **Impact:** Task 1 ActiveAgentsPanel component

**Hick's Law - Density Modes Insufficient**
- **Problem:** Compact mode still shows 24 items — Hick's Law applies to COUNT, not detail
- **Fix:** Default to "Active Brains Only" view (typically 3-5 running), show all 24 on-demand
- **Priority:** HIGH — affects time-to-completion for monitoring tasks
- **Impact:** Task 3 density mode implementation

**Swipe Gestures Discoverability**
- **Problem:** "Swipe left to reveal actions" not discoverable without visual hint
- **Fix:** Add first-run tooltip ("Swipe left for actions") OR subtle affordance (arrow peek at edge)
- **Priority:** HIGH — mobile primary interaction pattern
- **Impact:** Task 5 mobile swipe gestures

### CRITICAL Changes from Brain #4 (Frontend)

**RAF Batching Implementation**
- **Problem:** 24-brain burst needs RAF batching to maintain 60fps
- **Fix:** Wrap brain updates in startTransition, use useDeferredValue for brain list
- **Acceptance:** P99 frame time < 16.67ms during burst update
- **Impact:** Task 4 performance optimization

### HIGH Priority Changes from Brain #3 (UI Design)

**Status Badge Icons**
- **Problem:** Status badges (Gray/Green/Blue/Red) use color only
- **Fix:** Add icons (check circle, warning triangle, error X) alongside color
- **Acceptance:** Status badges have both color AND icon coding
- **Impact:** Task 2 StatusBadge component

**Ping Animation Easing**
- **Problem:** "CSS keyframes" mentioned but easing undefined
- **Fix:** Define ping animation with `cubic-bezier(0, 0, 0.2, 1)`
- **Impact:** Task 2 StatusBadge ping animation

**Density Mode Transition**
- **Problem:** "200ms" mentioned but easing not specified
- **Fix:** Use `ease-out` for mode switch (`cubic-bezier(0.215, 0.61, 0.355, 1)`)
- **Impact:** Task 3 density mode transitions

### HIGH Priority Changes from Brain #5 (Backend)

**WebSocket Batching Strategy**
- **Problem:** 24-brain burst = 24 simultaneous WebSocket messages (flood risk)
- **Fix:** Server-side aggregation into single "brain_batch" message
- **Acceptance:** WebSocket updates batched, client-side debouncing (100ms window)
- **Impact:** Task 5 WebSocket integration (needs backend spec)

## Implementation Notes

**RAF batching strategy:**
- Use React 18's automatic batching (no manual RAF needed)
- Wrap updates in startTransition for low-priority changes
- Use useDeferredValue for brain list (defer non-critical UI)

**Density mode implementation:**
- Use CSS grid with dynamic column count
- Compact: 6 columns (desktop), 2 columns (tablet), 1 column (mobile)
- Normal: 4 columns (desktop), 2 columns (tablet), 1 column (mobile)
- Detailed: 3 columns (desktop), 1 column (tablet), 1 column (mobile)

**Swipe gesture library:**
- Use native touch events (no library needed)
- Implement custom swipe handler (touchstart, touchmove, touchend)
- Fallback to button actions on desktop (hover to reveal)

**Performance optimization:**
- Virtualize if > 50 brains (react-window)
- For 24 brains, grid rendering is sufficient (no virtualization needed)
- Memoize BrainCard component (React.memo)

## Rollback Plan

If 60fps target not met:
1. Reduce concurrent brain updates to 12 (batch in 2 groups)
2. Disable ping animation (CSS animations can cause jank)
3. Use virtualization (react-window) for brain list

If swipe gestures fail on real devices:
1. Remove swipe gestures, keep button actions
2. Add bottom sheet for mobile actions (Material Design pattern)
3. Use react-swipeable library (fallback to native events)
