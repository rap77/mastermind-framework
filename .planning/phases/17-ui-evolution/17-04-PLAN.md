## Brain-Informed Updates

### CRITICAL Changes from Brain #5 (Backend)

**Cost Aggregation Strategy (CRITICAL)**
- **Problem:** O(n) aggregation on every dashboard load — performance issue
- **Required Fix:** Materialized view `cost_metrics_mv` (aggregates activity_log)
- **Performance SLA:** P50 < 10ms, P99 < 50ms (from materialized view)
- **Acceptance:** Cost dashboard loads in < 100ms end-to-end
- **Impact:** Task 1 costStore needs backend spec

### HIGH Priority Changes from Brain #2 (UX Research)

**QuotaBar Color-Only Coding**
- **Problem:** "Green/Yellow/Red" progress bar relies on color alone — WCAG violation
- **Fix:** Add percentage text AND icons (check/warning/error)
- **Priority:** MEDIUM — accessibility requirement
- **Impact:** Task 3 QuotaBar component

**Real-Time Updates Cognitive Overload**
- **Problem:** WebSocket streaming for 24 brains = continuous visual change
- **Fix:** Batch updates every 5s OR add "pause updates" button
- **Priority:** LOW — nice-to-have for focus-intensive tasks
- **Impact:** Task 5 WebSocket integration

### HIGH Priority Changes from Brain #5 (Backend)

**WebSocket Batching Strategy**
- **Problem:** 24-brain burst floods WebSocket connections
- **Fix:** Server-side rate limiting (max 5 batches/sec), client-side debouncing (100ms)
- **Acceptance:** WebSocket updates batched, connection errors trigger retry
- **Impact:** Task 5 WebSocket integration needs backend spec

### MEDIUM Priority Changes from Brain #3 (UI Design)

**QuotaBar Animation Easing**
- **Problem:** "200ms transition" needs easing function
- **Fix:** Use `ease-in-out` for value changes (`cubic-bezier(0.645, 0.045, 0.355, 1)`)
- **Impact:** Task 3 QuotaBar component

**Trend Indicator Color Coding**
- **Problem:** "↑ green for decreased cost, ↓ red for increased" is counterintuitive
- **Fix:** Use arrows to indicate cost direction (↑ = increased, ↓ = decreased)
- **Impact:** Task 2 MetricCard component

**Backend (CRITICAL - From Brain #5):**

**Rust Control Plane:**
- [ ] Create materialized view `cost_metrics_mv` (aggregates activity_log)
- [ ] Add trigger to refresh MV on every activity_log INSERT
- [ ] SQL:
  ```sql
  CREATE MATERIALIZED VIEW cost_metrics_mv AS
  SELECT
      brain_id,
      SUM(token_count) as total_tokens,
      SUM(duration_ms) / 1000 as total_duration,
      SUM(cost_usd) as total_cost,
      MAX(updated_at) as last_run_at
  FROM activity_log
  GROUP BY brain_id;

  REFRESH MATERIALIZED VIEW CONCURRENTLY cost_metrics_mv;
  ```
- [ ] Add caching layer (Redis) with 5s TTL

**Python API:**
- [ ] Expose query endpoint: `GET /api/costs/brains` (queries MV, not raw events)
- [ ] Performance SLA: P50 < 10ms, P99 < 50ms

## Implementation Notes

**WebSocket integration:**
- Reuse wsDispatcher from Phase 16 (Rust WebSocket Hub)
- Subscribe to `cost_updates` channel (real-time cost streaming)
- Parse JSON: `{ brainId, tokens, duration, cost, timestamp }`
- Batch updates via React.startTransition (low-priority)

**Budget enforcement:**
- Client-side only (visual warnings, no hard blocks)
- Server-side enforcement happens in Rust Control Plane (Phase 15)
- Yellow warning at 80% (soft limit)
- Red warning at 100% (hard limit, stop runs)

**Cost data source:**
- Read from Rust event sourcing (activity_log table)
- Python API aggregates cost per brain (GET /api/costs/brains)
- WebSocket pushes updates in real-time (Rust → Python → Frontend)

**Performance optimization:**
- MetricCard is pure component (React.memo)
- Cost metrics deferred via useDeferredValue
- WebSocket updates batched via startTransition

## Rollback Plan

If 60fps target not met:
1. Reduce cost update frequency to 5s (from real-time)
2. Batch cost updates in groups of 12 (2 commits instead of 1)
3. Remove trend indicators (reduce DOM updates)

If WebSocket integration breaks:
1. Fallback to polling (GET /api/costs/brains every 5s)
2. Show banner ("Real-time updates unavailable, using polling")
3. Add retry button to re-establish WebSocket connection
