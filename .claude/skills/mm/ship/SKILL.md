# mm-ship — Shipping & Versioning Skill

**Misión:** Ejecutar el flujo completo de shipping (verify → tag → archive → cleanup) cuando se completa un milestone o versión, asegurando que todas las precondiciones se cumplan antes de crear git tags y archivar artefactos.

## ¿Cuándo Usar?

Esta skill es **PROACTIVA** — se activa cuando:

**Al Completar Milestone:**
- El usuario indica que terminó un milestone ("terminé el milestone", "milestone completo")
- Todas las tareas de `tasks/plan.md` están marcadas como completas
- El usuario ejecuta `/mm:ship --verify` para verificar el estado
- Antes de crear un release o versión oficial

**Al Completar Versión:**
- El usuario quiere crear un tag semántico (vX.Y.Z)
- El usuario pregunta "¿cómo hago release?", "¿cómo versiono?"
- El usuario indica que quiere "shippear" lo que tiene
- Después de `/mm:complete-task` completa la última tarea de una fase

**Post-Implementation:**
- Después de implementar todas las features de un milestone
- Cuando el código está listo para producción
- Cuando se necesita archivar el trabajo completado

**No esperés a que el usuario lo pida explícitamente.** Si detectás alguno de estos indicios, activá la skill.

## Precondiciones (OBLIGATORIAS)

Antes de ejecutar el flujo de shipping, verificar:

### 1. Tests Pass ✅
```bash
# Backend tests (Python + pytest)
cd apps/api && uv run pytest
# Expected: 0 failures

# Frontend tests (Next.js + vitest)
pnpm --prefix apps/web test run
# Expected: 0 failures
```

**Si fallan:** Bloquear shipping y pedir que se corrijan los tests.

### 2. No Uncommitted Changes ✅
```bash
git diff --quiet
# Expected: exit code 0 (sin cambios sin commitear)

git diff --cached --quiet
# Expected: exit code 0 (sin cambios staged sin commitear)
```

**Si hay cambios:** Bloquear shipping y pedir que se commiteen primero.

### 3. SPEC.md Exists ✅
```bash
test -f tasks/SPEC.md
# Expected: exit code 0 (SPEC.md existe)
```

**Si no existe:** Bloquear shipping y pedir que se ejecute `/mm:discover` primero.

## Flujo de Shipping

### Paso 1: Verify (SIEMPRE primero)

Antes de hacer cualquier cambio, verificar el estado actual:

```bash
# Ejecutar verificación
/mm:ship --verify

# Output esperado:
MODE: verify
CURRENT_TAG: v0.2.2
NEXT_TAG: v0.2.3 (patch)
PRECONDITIONS:
  - Tests: ✅ PASS (631 backend + 407 frontend)
  - Uncommitted: ✅ CLEAN (0 files)
  - SPEC.md: ✅ EXISTS

CHANGELOG:
  abc1234 feat(phase-C): C2: Create review.md command with Brain integration
  def5678 fix(tests): update test_db_client.py to match real db_client.py API
  ghi9012 feat(phase-C): Complete /mm:review command with 5-axis code review

READY TO SHIP: ✅
```

**Si alguna precondición falla:**
```
PRECONDITIONS:
  - Tests: ❌ FAIL (3 failures in apps/api/tests/test_auth.py)
  - Uncommitted: ❌ DIRTY (5 files modified)
  - SPEC.md: ✅ EXISTS

READY TO SHIP: ❌ BLOCKED

Action required:
  1. Fix failing tests: cd apps/api && uv run pytest tests/test_auth.py
  2. Commit changes: /mm:safe-commit
  3. Run /mm:ship --verify again
```

---

### Paso 2: Tag (semántico)

Solo después de que `--verify` pase, crear el tag:

```bash
# Patch version (default) - v0.2.2 → v0.2.3
/mm:ship --patch

# Minor version - v0.2.2 → v0.3.0
/mm:ship --minor

# Major version - v0.2.2 → v1.0.0
/mm:ship --major

# Tag explícito
/mm:ship --tag v0.2.3
```

**El comando:**
1. Verifica precondiciones nuevamente
2. Lee el último tag: `git describe --tags --abbrev=0`
3. Calcula el siguiente tag (patch/minor/major)
4. Crea el tag: `git tag vX.Y.Z -m "changelog message"`
5. Muestra el changelog generado

**Output:**
```
MODE: tag
CURRENT_TAG: v0.2.2
NEXT_TAG: v0.2.3

PRECONDITIONS: ✅ ALL PASS

CREATING TAG: v0.2.3
git tag v0.2.3 -m "Release v0.2.3

Features:
- feat(phase-D): D1: Create ship-handler.py with verification logic
- feat(phase-D): D2: Create ship.md command documentation

Fixes:
- fix(handler): correct git describe command

Tests:
- test(coverage): 1038+ tests passing

Generated by MasterMind /mm:ship"

✅ Tag created: v0.2.3
```

---

### Paso 3: Archive (artefactos)

Después de crear el tag, archivar los artefactos:

```bash
# Archivar artefactos de la versión completada
/mm:ship --archive

# El comando:
# 1. Lee la versión del tag actual
# 2. Mueve tasks/ a .planning/archive/<version>/
# 3. Crea .planning/archive/<version>/summary.md
```

**Estructura de archivo:**
```
.planning/archive/
└── v0.2.3/
    ├── SPEC.md
    ├── plan.md
    ├── todo.md
    ├── summary.md
    └── task-progress.json
```

**Formato de summary.md:**
```markdown
# Milestone v0.2.3 — Archive Summary

**Released:** 2026-04-24
**Tag:** v0.2.3
**Previous:** v0.2.2

## What Was Shipped

### Features (3)
- feat(phase-D): D1: Create ship-handler.py with verification logic
- feat(phase-D): D2: Create ship.md command documentation
- feat(phase-D): D3: Create ship/SKILL.md with complete shipping protocol

### Fixes (1)
- fix(handler): correct git describe command to handle --abbrev=0 flag

### Tests (1038+)
- Backend: 631 tests passing
- Frontend: 407 tests passing

## Metrics

- **Duration:** 3 days
- **Commits:** 5
- **Files changed:** 14 new, 2 modified
- **Lines added:** 847
- **Lines removed:** 23

## Next Milestone

v0.3.0 — D4: ship-executor agent + D5: E2E validation

---

**Archived by:** MasterMind /mm:ship
**Archive path:** `.planning/archive/v0.2.3/`
```

---

### Paso 4: Cleanup (limpieza)

Finalmente, limpiar archivos temporales:

```bash
# Limpiar archivos temporales
/mm:ship --cleanup

# El comando:
# 1. Elimina .agent-*-running viejos
# 2. Elimina task-progress.json viejos
# 3. Actualiza .mastermind/config.yaml con nueva versión
```

**Archivos a limpiar:**
```bash
# Agent markers huerfanos
find .planning -name ".agent-*-running" -mtime +7 -delete

# Task progress viejos
find .planning -name "task-progress-*.json" -mtime +30 -delete

# Actualizar config
.mastermind/config.yaml:
  version: v0.2.3
  last_shipped: "2026-04-24T15:30:00Z"
```

**Output:**
```
MODE: cleanup

CLEANING:
  - Removing old agent markers: 3 files deleted
  - Removing old task progress: 1 file deleted
  - Updating .mastermind/config.yaml

✅ Cleanup complete
```

## Fixes

## Tests

## Docs

## Formato de Archive

Los artefactos se archivan en `.planning/archive/<version>/`:

```
.planning/archive/
└── v0.2.3/
    ├── SPEC.md              # Especificación del milestone
    ├── plan.md              # Plan de implementación
    ├── todo.md              # Checklist de tareas
    ├── summary.md           # Resumen del milestone
    └── task-progress.json   # Progreso de tareas (snapshot)
```

**Reglas de archivo:**
1. Solo archivar después de crear tag
2. Mantener todos los archivos de `tasks/`
3. Crear `summary.md` con métricas
4. Guardar snapshot de `task-progress.json`
5. No archivar si hay tareas pendientes

## Integración con /mm:ship

Esta skill se activa cuando el comando `/mm:ship` se ejecuta:

```bash
# El comando ejecuta:
/mm:ship --verify     # Solo verificar precondiciones
/mm:ship --patch      # Incrementar patch (v0.2.2 → v0.2.3)
/mm:ship --minor      # Incrementar minor (v0.2.2 → v0.3.0)
/mm:ship --major      # Incrementar major (v0.2.2 → v1.0.0)
/mm:ship --tag v0.2.3 # Tag explícito
/mm:ship --archive    # Solo archivar artefactos
/mm:ship --cleanup    # Solo limpiar temporales

# El flujo completo:
1. /mm:ship --verify      # Verificar que todo esté OK
2. /mm:ship --patch       # Crear tag
3. /mm:ship --archive     # Archivar artefactos
4. /mm:ship --cleanup     # Limpiar temporales
```
